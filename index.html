Explainer 01 — the context window

Context is a resource, not a memory.

An agent's context window is its working memory — a scratchpad of fixed width. For today's models that's about 200k tokens, and up to a million at the very frontier. The bar to the right has a hard edge, and you can watch it fill.

As it fills, the oldest material doesn't vanish at a cliff — it fades. Coherence degrades as a gradient ("context rot"). The fix is compaction: crush the old turns into a dense summary — a sedimentary layer — and reclaim room. Each compaction lays down a small layer that sticks around; compact again and a fresh layer forms beside it — the older ones aren't re-crushed.

Explainer 02 — agent skills

Progressive disclosure.

A skill is just a Markdown file: a tiny header (a name and a description — the trigger) over a much larger body (the actual procedure, sometimes with runnable scripts). Only the headers load up front.

So you can keep dozens of skills on the shelf for almost nothing — add as many as you like and the budget barely moves. When a task arrives that matches, just that one body unfolds and is paid for. N headers ≈ free; one body = the cost, and only on a match. That asymmetry is the whole trick.

Explainer 03 — orchestration ⇄ skills

The same work, in space or in time.

Anything a team of agents does by passing messages, a single agent can do by swapping hats in sequence. A factory line spreads the work across space — many fixed-role agents at once. A one-actor play spreads the same work across time — one agent, many hats, one after another.

Orchestration and skills are the two ends of that one spectrum. The only real difference is parallelism.
